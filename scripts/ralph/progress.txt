  - `verifyPaymentChannelClaim` throws on malformed/invalid hex signatures — always wrap in try/catch and treat exceptions as `valid = false`
  - These are pure crypto operations — no XRPL network connection needed at all

## Codebase Patterns
- PaymentChannel ledger entry type in AffectedNodes metadata is "PayChannel" NOT "PaymentChannel"
- xrpl.js `isCreatedNode(n)` helper from "xrpl" is the correct way to narrow Node union type
- Use `TransactionMetadataBase` (not `TransactionMetadata`) for generic channel/offer transactions
- resolveWallet pattern is duplicated per command file (no shared util yet) — follow the same shape
- XRPL epoch conversion: Math.floor(new Date(iso).getTime()/1000) - 946684800
