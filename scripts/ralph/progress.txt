## Codebase Patterns
- Ticket-based parallel funding: use `initTicketPool` + `nextTicket()` + `Sequence:0, TicketSequence:n` to fund N wallets concurrently from master without sequence conflicts
- Each `it()` block must call `createFunded` to get fresh wallets — never share across test cases
- `tests/e2e/helpers/fund.ts` exports: `XRPL_WS`, `fundMaster`, `initTicketPool`, `nextTicket`, `generateWallets`, `createFunded`
- Existing testnet helper at `tests/helpers/testnet.ts` exports `fundFromFaucet` (1 faucet per call) and `TESTNET_URL`
- `fundAddress(client, master, address, amountXrp)` in fund.ts funds a specific wallet address (e.g. mnemonic-derived) using a ticket — use when `createFunded` (random wallets) won't work
- For tests needing an active client (xrpl.js calls inside it()), keep `client` as file-level variable connected in beforeAll / disconnected in afterAll
- Fund wallets with 25-30 XRP (not the default 10) — testnet base reserve is 10 XRP, leaving 0 available for transactions with the 10 XRP default
- Use a local helper function (e.g. `setupMakerIssuer()`) within the test file to DRY up repeated wallet+trustline setup, while still creating fresh wallets per test
