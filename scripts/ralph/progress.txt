- Fixed `issuanceGetCommand` to display the `issuanceId` argument (24-byte) as the MPTokenIssuanceID label, not `entry.index`
- Tests cover: list text output with ID, list --json, list empty account ("No MPT issuances."), get text output with all properties, get --json
- Files changed: `tests/e2e/mptoken/mptoken.query.test.ts` (new), `src/commands/mptoken.ts` (bug fix), `scripts/ralph/prd.json`
- **Learnings for future iterations:**
  - `MPTokenIssuance.index` from `account_objects` / `ledger_entry` is a 32-byte hash key, NOT the 24-byte MPTokenIssuanceID
  - To get the 24-byte ID from an `account_objects` entry: use `Sequence` (present in rippled response, not in xrpl.js types) + `Issuer` via `computeIssuanceId()`
  - For `get` command: display the issuanceId argument (user-provided 24-byte ID), not `entry.index`
  - The `--json` output of `list` and `get` returns raw ledger data with 32-byte `index`; tests for `--json` should match by `Issuer`/`AssetScale` not by `index`

## 2026-03-07 - US-003
- Implemented `tests/e2e/mptoken/mptoken.authorize.test.ts` with 7 tests covering all authorize command options
- The `authorize` command was already implemented in `src/commands/mptoken.ts`; only the E2E tests were missing
- Tests cover: holder opts in, holder opts out (--unauthorize), issuer authorizes holder on require-auth issuance (--holder), issuer revokes (--holder --unauthorize), --json, --dry-run, --no-wait
- Files changed: `tests/e2e/mptoken/mptoken.authorize.test.ts` (new), `scripts/ralph/prd.json`
- **Learnings for future iterations:**
  - CRITICAL: `MPTokenIssuanceID` in transactions is 24 bytes (4-byte Sequence BE + 20-byte AccountID) — NOT the 32-byte `LedgerIndex` from `AffectedNodes`. Use the Sequence+AccountID formula (same as `mptoken.issuance.test.ts`) when creating issuances directly via xrpl.js in tests.
  - The CLI's LedgerIndex fallback in `mptoken.ts` would produce a wrong 32-byte value; in practice `meta.mpt_issuance_id` is always present on testnet so the fallback is never hit.
  - For require-auth issuances: holder must opt in FIRST (MPTokenAuthorize from holder), then issuer authorizes (MPTokenAuthorize from issuer with Holder field). Both steps are needed before the holder can receive tokens.
  - `fundFromFaucet` is the only faucet call pattern in this project (testnet.ts); there is no `fund.ts` helper with ticket pools.
